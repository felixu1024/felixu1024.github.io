引言

其实工厂方法模式就好比是造车，在没有工厂的时候，我们想要一辆车怎么办呢，只能是亲力亲为，自己造呗，到java上就是new实例嘛。那有了工厂呢，我们想要一辆车，我们想要一辆车，我们找个4s店，告诉他，我想要一辆奇瑞车，他说好，然后他找到奇瑞工厂，生产了一辆奇瑞车给你。又一天，你想换辆车，你又来了，你说这次我要辆奔驰吧，然后他又找了奔驰工厂，给你造了一辆奔驰。这就是我们的工厂方法模式。

定义

Define an interface for creating an object,but let subclasses decide which class to instantiate.Factory Method lets a class defer instantiation to subclasses.
意思是：定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法是一个类的实例化延迟到其子类。

以引言为例，类图如下：

Product:具体产品类的抽象，可以是接口或者抽象类

Benz:继承自产品抽象的具体产品

BMW:继承自产品抽象的具体产品

Factory:抽象工厂，他是工厂模式的核心，具体创建过程由其子类实现

CreateCarFactory:具体工厂，继承自抽象工厂，实现了具体的创建过程

client:在本例中就是客户，也就是场景类调用工厂产生实例

工厂方法模式的优点

  1. 优秀的拓展性，比如客户要再买一辆Lamborghini，我们什么都不需要改
  2. 良好的封装性
  3. 屏蔽具体产品类
  4. 解耦

    使用场景

    既然工厂方法模式有这么多的好处，那我们是不是就要经常去使用它呢，也不完全是这样的，因为通过上面的案例也可以看到，它相比我们直接new对象，明显多了一些类，增加了代码的复杂度。所以我们也需要考虑它的使用场景：
  5. 需要灵活可扩展的框架时，比如JDBC连接数据库，我们可能用Mysql也可能用Oracle，只需要切换驱动就行了
  6. 当调用方不需要知道对象创建过程

    工厂方法模式的扩展

  7. 简单工厂
    该模式是工厂方法模式的弱化，因为简单，所以叫做简单工厂模式，也叫做静态工厂模式。它就是去掉了抽象工厂，把创建实例对象的方法变为静态方法。但它其实是不符合开闭原则的，不难想象，这样的工厂其实是很难去扩展的，当然，在产品部分它是符合开闭原则的。
  8. 多工厂类
  9. 替代单例
  10. 延迟初始化
