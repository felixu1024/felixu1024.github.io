23种设计模式之抽象工厂模式

felixu | 发表于 2018-04-03 | 分类于 Design Pattern |
Abstract Factory Pattern

引言

前面讲了工厂方法模式、简单工厂，这里接着讲讲抽象工厂模式
我们知道了，工厂方法模式，其实就是屏蔽了产生对象的细节，而只对调用需要的结果负责
那么抽象工厂模式又是什么呢，又和工厂方法模式有什么关系呢

定义

Provide an interface for creating families of related or dependent objects without specifying their concrete classes.（为创建一组相关或相互依赖的对象提供一个接口，而且无需指定它们的具体类）

从定义可以看出，他有个组的概念，那么该怎么来理解呢。其实抽象工厂模式是工厂方法的升级，工厂方法模式是针对一个产品的，而抽象工厂是针对一组产品
在前面的工厂方法模式中，我们以汽车来说明的，这里同样继续以汽车为例来叙述
产品衍生自一个接口或抽象类，即我们的汽车接口
在工厂方法模式中，只有一个车接口，对应着有不同的实现，比如奔驰，宝马等
那么在抽象工厂模式中呢，产品作为最顶层接口，抽象出不同的产品族，同一产品族下有对应产品实现
创建工厂，由顶层抽象，对于产品分类的具体创建有不同的具体实现
这样说起来可能是有些抽象，下面用代码来描述

代码描述

产品的顶层接口 

public interface Car {

    /**
     * 汽车名称
     * @return
     */
    String getName();

    /**
     * 汽车品牌
     * @return
     */
    String getType();
}

宝马系 

public abstract class AbstractBmw implements Car {

    /**
     * 汽车品牌
     * @return
     */
    public String getType() {
        return "我是宝马牌";
    }
}

奔驰系 

public abstract class AbstractBenz implements Car {

    /**
     * 汽车品牌
     * @return
     */
    public String getType() {
        return "我是奔驰牌";
    }
}

宝马车实现 

public class BmwA extends AbstractBmw {

    /**
     * 汽车名称
     * @return
     */
    public String getName() {
        return "我是小汽车";
    }
}

public class BmwB extends AbstractBmw {

    /**
     * 汽车名称
     *
     * @return
     */
    public String getName() {
        return "我是SUV吧";
    }
}

奔驰也同样实现这样两个具体的车，这里，就不重复了
然后是抽象工厂

public abstract class AbstractFactory {

    /**
     * 创建宝马系
     * @return
     */
    public abstract AbstractBmw createBmw();

    /**
     * 创建奔驰系
     * @return
     */
    public abstract AbstractBenz createBenz();
}

具体工厂实现 

public class ProductAFactory extends AbstractFactory {

    /**
     * 创建宝马系
     *
     * @return
     */
    @Override
    public AbstractBmw createBmw() {
        return new BmwA();
    }

    /**
     * 创建奔驰系
     *
     * @return
     */
    @Override
    public AbstractBenz createBenz() {
        return new BenzA();
    }
}

public class ProductBFactory extends AbstractFactory {

    /**
     * 创建宝马系
     *
     * @return
     */
    @Override
    public AbstractBmw createBmw() {
        return new BmwB();
    }

    /**
     * 创建奔驰系
     *
     * @return
     */
    @Override
    public AbstractBenz createBenz() {
        return new BenzB();
    }
}

可以看到的是，在这里，我们把同一类别的车放到了同一个工厂中，即将小轿车放在一起，将SUV放在一起了
测试代码

public class TestAbstr {

    public static void main(String[] args) {
        AbstractFactory factoryA = new ProductAFactory();
        AbstractFactory factoryB = new ProductBFactory();
        // 想要一辆宝马的SUV
        AbstractBmw bmw = factoryB.createBmw();
        System.out.println(bmw.getName() + " " + bmw.getType());
        //想要一辆奔驰的小轿车
        AbstractBenz benz = factoryA.createBenz();
        System.out.println(benz.getName() + " " + benz.getType());
    }
}

测试结果

我是SUV吧 我是宝马牌
我是小轿车 我是奔驰牌

这就是抽象工厂模式的实现过程啦，结合之前对于产品族的解释，应该会更好理解一点吧

优缺点

优点：

  1. 封装性，不需要关心对象如何创建，屏蔽细节
  2. 产品族内约束，隐式的对产品族内进行约束
    缺点：
    产品族的扩展会很难，你想，如果加入一个新的产品族，那代码需要修改很多。已经违反了开闭原则

    使用场景

    当需要创建的对象是一系列相互关联或相互依赖的产品族时，便可以使用抽象工厂模式。说的更明白一点，就是一个继承体系中，如果存在着多个等级结构（即存在着多个抽象类），并且分属各个等级结构中的实现类之间存在着一定的关联或者约束，就可以使用抽象工厂模式。假如各个等级结构中的实现类之间不存在关联或约束，则使用多个独立的工厂来对产品进行创建，则更合适一点。

    总结

    无论是简单工厂模式，工厂方法模式，还是抽象工厂模式，他们都属于工厂模式，在形式和特点上也是极为相似的，他们的最终目的都是为了解耦。在使用时，我们不必去在意这个模式到底工厂方法模式还是抽象工厂模式，因为他们之间的演变常常是令人琢磨不透的。经常你会发现，明明使用的工厂方法模式，当新需求来临，稍加修改，加入了一个新方法后，由于类中的产品构成了不同等级结构中的产品族，它就变成抽象工厂模式了；而对于抽象工厂模式，当减少一个方法使的提供的产品不再构成产品族之后，它就演变成了工厂方法模式。
    所以，在使用工厂模式时，只需要关心降低耦合度的目的是否达到了
